## Context

本项目是基于 Taro + React + TypeScript 的工具型小程序集合，采用多页面结构承载多个轻量功能（如手持弹幕、做决定、二维码、节拍器、拼豆图、猜数字等）。当前代码可运行，但缺少统一的产品与技术基线文档，新增需求时需要反复“读代码找上下文”，导致需求澄清、影响评估、测试设计和实施排期效率偏低。

当前约束与现状：
- 入口结构以 `src/pages/classify` 聚合，能力入口依赖 `classify/constants.ts` 与 `app.config.ts` 的页面注册。
- 功能实现分散在 `src/pages/**`，复用能力主要在 `src/components/**`、`src/utils/**`、`src/hooks/**`。
- 平台能力调用（如相册权限、Canvas、文件系统、分享）直接分布于页面与工具层，尚无系统化“能力矩阵”文档。
- 工程配置集中于 `config/**` 和 `package.json`，依赖与构建策略未沉淀到技术说明中。

主要干系人：产品/需求提出方、功能开发者、测试人员、后续接手维护者。

## Goals / Non-Goals

**Goals:**
- 建立可持续维护的 PRD 基线，明确目标用户、核心场景、功能地图、优先级和非功能要求。
- 建立可执行的技术基线，明确架构分层、关键模块、平台依赖、构建发布和质量约束。
- 将文档和代码资产建立一一映射，降低后续新增需求前的认知成本。
- 提供文档更新策略，使后续迭代可增量维护而非重写。

**Non-Goals:**
- 不在本变更中重构现有业务代码或替换技术栈。
- 不引入新的运行时依赖或构建插件。
- 不在本阶段输出单个新功能的详细交互稿/接口定义。

## Decisions

1. 采用“双文档基线”而非单文档
- Decision: 分别维护 PRD 基线与技术基线，避免产品语义和实现细节耦合。
- Rationale: 使用方不同，变更频率不同；分离后更利于并行维护和审阅。
- Alternatives considered:
  - 单文档承载全部内容：结构冗长，更新冲突多。
  - 仅技术文档：无法支持需求侧快速评估。

2. 以“现有代码反向建模”为主要信息来源
- Decision: 文档内容以 `app.config.ts`、`pages/**`、`utils/**`、`config/**` 的现状分析为准。
- Rationale: 当前缺少历史需求文档，代码是唯一可信来源。
- Alternatives considered:
  - 仅基于口头需求撰写：容易与真实实现偏离。
  - 先大规模补测试再写文档：投入高，无法快速提供迭代基线。

3. 文档采用“能力边界 + 迭代建议”结构
- Decision: 每个功能模块在 PRD 中定义场景、输入/输出、限制与优化方向；在技术文档中定义模块职责、依赖点和风险点。
- Rationale: 直接服务“新增需求评估”场景，减少从零分析成本。
- Alternatives considered:
  - 只做目录级说明：无法支撑需求拆解和测试范围判断。

4. 将文档维护纳入流程规则
- Decision: 新增页面、改动关键交互或增加平台能力调用时，要求同步更新 PRD 与技术文档对应章节。
- Rationale: 防止文档一次性产出后失效。
- Alternatives considered:
  - 依赖人工自觉：长期大概率失真。

## Risks / Trade-offs

- [风险] 代码已存在但行为未被测试覆盖，文档可能遗漏边界条件  
  → Mitigation：在文档中明确“已知限制/待验证项”，并将验证任务写入后续 tasks。

- [风险] 功能较多且分散，首次整理成本高  
  → Mitigation：先建立基线版本，后续按功能迭代增量更新。

- [风险] 文档粒度过细导致维护负担上升  
  → Mitigation：限定为“需求评审可用”的必要信息，不记录实现级细枝末节。

- [trade-off] 当前不做代码重构，短期可快速交付文档，但历史技术债仍存在  
  → Mitigation：在技术文档中显式记录技术债与建议优先级。

## Migration Plan

1. 在本 change 中完成 proposal/design/specs/tasks，形成 apply-ready 的实施清单。  
2. 按 tasks 执行时，先输出 PRD 文档初版，再输出技术文档初版。  
3. 通过一次团队评审确认文档结构与覆盖范围。  
4. 将文档更新触发条件纳入后续需求流程（如需求评审 checklist）。  

回滚策略：
- 如文档结构不被接受，可保留 change 目录并调整 specs 与 tasks，不影响运行时代码与发布流程。

## Open Questions

- PRD 文档最终落地位置是否固定在仓库根目录（如 `docs/prd.md`）还是按版本拆分（如 `docs/prd/v1.md`）？
- 技术文档是否需要补充“页面级时序图/流程图”以支持新人 onboarding？
- 后续是否需要将 OpenSpec 文档与 CI 检查联动（如关键路径变更必须更新文档）？
