## ADDED Requirements

### Requirement: PRD 基线文档覆盖核心产品信息
系统 MUST 提供一份项目级 PRD 基线文档，完整描述产品目标、用户场景、功能地图、优先级与非功能要求，用于后续需求迭代前的统一对齐。

#### Scenario: 生成 PRD 基线初版
- **WHEN** 团队发起“为现有 Taro 小程序建立产品基线文档”的变更
- **THEN** 文档 SHALL 包含产品定位、目标用户、典型使用场景、页面能力清单与业务边界

#### Scenario: PRD 包含迭代评估所需信息
- **WHEN** 团队评估新增功能需求
- **THEN** 文档 MUST 能支撑范围评估、优先级判断和回归影响识别

### Requirement: PRD 文档需要与现有能力一一映射
系统 SHALL 要求 PRD 中每项核心能力都可映射到现有页面或功能入口，避免脱离代码现状的抽象描述。

#### Scenario: 功能映射可追踪
- **WHEN** 评审者查看某项能力描述
- **THEN** 文档 MUST 给出对应页面或入口标识（例如分类页入口与目标页面路径）

#### Scenario: 发现文档与实现不一致
- **WHEN** 发现 PRD 描述的能力在代码中不存在或行为不一致
- **THEN** 变更 SHALL 标记该项为“待校准”，并在任务列表中补充核查动作

### Requirement: PRD 文档必须定义维护机制
系统 MUST 为 PRD 文档定义更新触发条件和最小维护粒度，保证后续迭代中持续有效。

#### Scenario: 新增功能触发更新
- **WHEN** 新增页面或新增核心交互能力进入开发
- **THEN** 团队 MUST 同步更新 PRD 的功能地图与优先级说明

#### Scenario: 需求调整触发更新
- **WHEN** 现有功能的目标用户、主流程或关键约束发生变化
- **THEN** PRD SHALL 在同一迭代内完成更新并记录变更摘要
